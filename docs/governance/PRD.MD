# BP-CMS-Scraper PRD

## 1) 核心目标
- 在尽量不改动现有 CMS/BP/FB 连接方式的前提下，实现“抓取 -> 评分 -> 排程 -> 人工审核 -> 发布/回写日志”的自动化流水线。
- 将原 WhatsApp 审核流升级为 Telegram Bot 审核流，并提供 Streamlit 中控界面。
- 支持 GCP VM + cronjob 的稳定定时执行，以及故障告警和可回放日志。

## 2) 功能范围（In Scope）
- CMS 娱乐板块抓取（后续扩展到更多板块），提取标题、时间、reached、engaged 等字段。
- 评分管线：规则评分 + LLM 评分融合，产出可排序候选列表。
- 排程引擎：支持时段策略、冲突规避、post 类型约束（photo/text）。
- 人工介入：Telegram Bot 审核（通过/否决/微调），超时自动确认。
- 可视化与日志：Streamlit 控制台查看任务状态、运行日志、错误详情。
- 数据层：Firebase Storage 保存日志与处理结果，后续可扩展到 Firestore/RTDB 索引。
- 运维：cron 定时任务、告警、升级脚本（拉代码/安装依赖/重启服务）。

## 3) 非范围（Out of Scope）
- 矩阵账号体系相关能力（邮箱注册、AdsPower 实例轮换、多社媒建号、暖机矩阵、跨平台成效统计）。
- 对现有 CMS/BP/FB 组织架构做侵入式改造。

## 4) 成功指标（MVP）
- 抓取任务成功率 >= 95%（按日统计）。
- 评分任务按时完成率 >= 95%（在配置窗口内产出候选）。
- Telegram 审核消息送达率 >= 99%，超时自动确认可配置。
- 排程冲突率 <= 1%（同时间槽冲突或重复发布）。
- 告警平均通知延迟 <= 2 分钟。

## 5) 验收清单
- 任务链路可全程追踪：每个任务均有 `run_id` 与状态迁移记录。
- 关键失败场景有明确处理：重试、降级、人工接管。
- 中控台可查看近 7 天任务日志与失败明细。
- 升级脚本可在不中断核心任务的窗口内执行滚动升级。
